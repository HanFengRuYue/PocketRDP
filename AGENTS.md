## Native (FreeRDP) build

**Active via prebuilt jniLibs вҖ?FOUR ABIs.** `core-rdp/src/main/jniLibs/{arm64-v8a,armeabi-v7a,x86,x86_64}/` each ship **9** .so files compiled in WSL2 Ubuntu: `libfreerdp-android` + `libfreerdp3` + `libfreerdp-client3` + `libwinpr3` + `libssl` + `libcrypto` + `libcjson` + `liburiparser` + `libc++_shared`. So the APK is a single fat binary carrying all four architectures (вү?Г— the native payload of the old arm64-only build). There is **no** `libopenh264.so` вҖ?H.264 comes from **FFmpeg statically linked into `libfreerdp3.so`** (which is why it's ~18вҖ?3 MB per ABI). **No comment-region toggle anymore:** `core-rdp/build.gradle.kts` is now **property-gated** вҖ?a normal Windows build (`./gradlew :app:assembleDebug`, no extra props) just packages the four prebuilt ABIs (no native toolchain needed); passing **`-PnativeAbi=<abi>`** turns the FreeRDP CMake superbuild ON for that *one* ABI (WSL2 only). The driver that loops the ABIs is `scripts/build-native-multiarch-in-wsl.sh` (see Native build вҶ?multiarch below). **`libcjson.so` and `liburiparser.so` are mandatory** вҖ?they are `DT_NEEDED` of `libwinpr3.so`; without them `dlopen(libfreerdp-android.so)` fails on the recursive dependency resolve. The other **8** libs (per ABI) must stay in `packaging { jniLibs { pickFirsts } }` so AGP doesn't drop them at merge time (the pickFirsts list is generated for all four ABIs Г— 8 libs by a Kotlin loop); `libfreerdp-android.so` itself is NOT in pickFirsts (single JNI entry, no duplicate merge source) and doesn't need to be.

**Prebuilt .so feature set**: `WITH_MEDIACODEC=ON` (forced in `ExternalFreeRDP.cmake`) + `WITH_FFMPEG=ON` + `WITH_OPENH264=OFF` (+ `WITH_VIDEO_FFMPEG=ON`, `WITH_DSP_FFMPEG=OFF`); `WITH_OPUS/WEBP/JPEG/PNG=OFF`. **DECODER REALITY (corrected 2026-06 вҖ?earlier notes here said "FFmpeg software is the only H.264 subsystem", which is WRONG and misled a latency audit):** `h264.c` registers subsystems **MediaCodec вҶ?OpenH264(absent) вҶ?FFmpeg** and takes the first whose `Init()` succeeds, so on a real ARM device the **PRIMARY decoder is the Android hardware MediaCodec** (`libfreerdp3.so` carries `DT_NEEDED libmediandk.so` + `AMediaCodec_*`, with a PocketRDP local low-latency patch in `h264_mediacodec.c`: `AMediaFormat_setInt32(format,"low-latency",1)`). The statically-linked **FFmpeg software decoder + swscale is the FALLBACK** (emulators / devices whose HW codec init fails; its decode is single-threaded by default вҖ?see CODEC-3 below). Either way `LibFreeRDP.hasH264Support()` returns true and `RdpClient.buildCommandLine` emits **`/gfx:AVC444`** (з”»иҙЁдјҳе…Ҳ, full 4:4:4 chroma) or **`/gfx:AVC420`** (жөҒз•…дјҳе…Ҳ, single YUV420 stream вҖ?~half the software-decode cost, no `prim_YUV444` recombine, lower control latency) when `useH264`, chosen by `entity.preferAvc420` (DB v6, default false = AVC444); else `/gfx:RFX`. Never both an AVC mode and RFX вҖ?they conflict. AVC420 is also clean with FFmpeg (the magenta/green grid was an OpenH264-only recombine-stride bug, and AVC420 has no recombine). It also always emits `/multitouch` for the rdpei channel, and `/drive:PocketRDP,<path>` when file redirection is on (see Settings).
  - **Why FFmpeg, not OpenH264 (the еҚЎйЎҝвҶ’иҠұеұ?saga):** OpenH264 was tried first (WITH_OPENH264=ON, runtime-dlopen mode) and fixed the video еҚЎйЎҝ against the `AVC444ModePreferred=1` host, but `/gfx:AVC444` rendered a **diagonal magenta/green chroma grid over an otherwise-correct (luma-intact) image** вҖ?FreeRDP recombines AVC444's dual streams into YUV444 via its own `prim_YUV`, and OpenH264's decoded plane strides mis-fed that combine. `/gfx:AVC420` (single YUV420 stream, no combine) was clean on OpenH264 but only 4:2:0. Switching the decoder to **FFmpeg** (whose decoded-frame strides feed the combine correctly) made **AVC444 render cleanly, field-confirmed** вҖ?so we keep AVC444.
  - **FFmpeg is STATIC** (`ExternalFFmpeg.cmake`: `--disable-shared --enable-static`, minimal `--enable-decoder=h264 --enable-parser=h264` + swscale, no opus/openh264/audio/mediacodec). A shared FFmpeg bakes versioned SONAMEs (`libavcodec.so.61`) into `libfreerdp3` as `DT_NEEDED`, which Android can't package; static links it straight into `libfreerdp3.so` (PIC), no extra `.so`, no soname problem, and 16 KB alignment comes for free from `libfreerdp3`'s own link flags. FreeRDP is told to prefer it via `-DWITH_VIDEO_FFMPEG=ON` in `ExternalFreeRDP.cmake`.
  - **FFmpeg source is NOT vendored** (unlike openh264) вҖ?`ExternalFFmpeg.cmake` would download `n8.1.tar.gz` from GitHub. `scripts/fetch-ffmpeg-src.sh` pre-stages it into `cpp/external/ffmpeg/` (with mirror fallbacks) so the WSL build never needs the network mid-build. Run it before re-building native.
  - If FFmpeg ever has to go (size/CPU), the fallback is `WITH_OPENH264=ON` + `WITH_FFMPEG=OFF` + `/gfx:AVC420` (re-add `libopenh264.so` to jniLibs/pickFirsts and the `WITH_OPENH264_LOADING=ON` line in ExternalFreeRDP.cmake вҖ?both still present in git history).

Desktop scale uses **`/scale-desktop:<100..300>`** (`FreeRDP_DesktopScaleFactor`, continuous) in `buildCommandLine()` вҖ?NOT `/scale`. `/scale` is hard-limited by `parse_scale_options` to {100,140,180}, so the old code quantised to those buckets and every slider value >180 silently collapsed to 180 (field bug: зј©ж”ҫж»‘еқ—еҸӘжңүдёүжЎЈзңҹжӯЈз”ҹж•Ҳ). `/scale-desktop` honours the exact value (it accepts 100..500; we `coerceIn(100,300)` to the slider range), so the 100вҖ?00 % slider has real continuous effect. It sets DesktopScaleFactor only (not DeviceScaleFactor like /scale did) and is the **remote** desktop's DPI/font scaling, not local picture zoom. Both are INTEGER options (value required) вҖ?note the BOOL-option trap below is a *different* class. When `parse_arguments` fails the snackbar reads "freerdp_parse_arguments failed: Success" because `last_error_code` is only populated during `freerdp_connect`, not during argument parsing вҖ?look at the redacted `Connecting with args:` logcat line instead.

**FreeRDP BOOL-option disable trap.** Our args use the slash/colon syntax (`COMMAND_LINE_SEPARATOR_COLON | SIGIL_SLASH | SIGIL_PLUS_MINUS`, no SEPARATOR_SPACE/ENABLE_DISABLE). For a `COMMAND_LINE_VALUE_BOOL` option, *not adding the flag leaves FreeRDP's default* вҖ?which for several options is ON. To force-disable you must emit the **`-opt`** form (the `-` sigil вҶ?`BoolValueFalse=NULL` вҶ?handler `enable=FALSE`). **`/opt:off` FAILS** with `COMMAND_LINE_ERROR_UNEXPECTED_VALUE` (a BOOL option rejects a trailing value), aborting the whole connect. This is why `useMultitransport=false` emits `-multitransport`, not `/multitransport:off` (see the UDP bullet).

**Native touch + transport JNI (added 2026-05; expanded 2026-06).** `LibFreeRDP.java` exposes `freerdp_send_touch(inst, contactId, x, y, action)`, `freerdp_get_transport_info(inst)`, and now `freerdp_get_transport_stats(inst)`. The touch side: `android_freerdp.h` `androidContext` holds a `RdpeiClientContext* rdpei` (stored on `RDPEI_DVC_CHANNEL_NAME` connect, cleared on disconnect); `android_event.{h,c}` add `EVENT_TYPE_TOUCH` вҶ?`android_process_event` calls `rdpei->TouchBegin/Update/End` and forces `rc = TRUE` (a dropped touch when the channel isn't up must NOT tear down the session). The transport side now delegates to FreeRDP core state instead of guessing from capability flags: `freerdp_get_transport_info` returns bits `0..3` = actual state (`TCP`, `UDP-R`, `UDP-L`, `UDP2`), bit 8 = UDP requested but fell back to TCP, bit 9 = multitransport requested, and bits `4..7` = selected security protocol; `freerdp_get_transport_stats` returns `[inBytes, outBytes, inPackets, outPackets, retransmits, failureStage, tunnelHr, socketError]` for the status dropdown. **If you change these signatures, re-run the WSL build** вҖ?they live in the prebuilt `.so`. **Unicode input force-enable (2026-06).** `android_post_connect` now force-sets `FreeRDP_UnicodeInput=TRUE` right after the capability exchange so CJK/IME text actually reaches the server even when the host doesn't advertise `INPUT_FLAG_UNICODE` (see the TextInputEncoder bullet for the full why). It's a one-line change in `android_freerdp.c` вҶ?only `libfreerdp-android.so` changes, so it rebuilt cleanly via `scripts/build-native-multiarch-in-wsl.sh` with all four ABIs passed explicitly (the script default skips arm64-v8a). The OpenSSL/FFmpeg/FreeRDP deps stayed cached, so each ABI was just a freerdp-android recompile + relink вҖ?the slow part was FreeRDP's own CMake reconfigure on the `/mnt/d` 9p mount (~15 min for the first ABI's `try_compile` sweep), NOT a dep rebuild; the later ABIs reused enough cache to finish in ~6 min each.

**16 KB page-size is mandatory for the 64-bit ABIs** (arm64-v8a, x86_64) on Android 15+ (SDK 35+) devices and the `sdk_gphone16k_*` emulators вҖ?dlopen rejects any 64-bit .so whose LOAD segments are 4 KB-aligned with `program alignment (4096) cannot be smaller than system page size (16384)`. NDK 27.1 only injects `-Wl,-z,max-page-size=16384` for the *cmake-built* libs (freerdp3/winpr3/freerdp-client3/cjson/uriparser) when `ANDROID_SUPPORT_FLEXIBLE_PAGE_SIZES=ON` is passed to cmake, AND ExternalProject children don't inherit init linker flags so each child needs the flag explicitly forwarded. The fix lives in three places and **must** stay synced:
1. `core-rdp/build.gradle.kts`: cmake `arguments` block inside the `if (nativeAbi != null) { ... }` native-build gate (active only under `-PnativeAbi=<abi>`).
2. `third_party/FreeRDP/client/Android/Studio/freeRDPCore/src/main/cpp/CMakeLists.txt`: `ANDROID_CMAKE_ARGS` forwards the flags to cjson/uriparser/openssl ExternalProjects (local patch on top of the pinned submodule).
3. `third_party/FreeRDP/client/Android/cmake/ExternalFreeRDP.cmake`: `FREERDP_LINKER_FLAGS` includes `-Wl,-z,max-page-size=16384` because that line explicitly overrides `CMAKE_SHARED_LINKER_FLAGS` for the freerdp3/freerdp-client3/winpr3 child build.

**Per-ABI alignment reality (verified 2026-06 against the shipped APK):** `libssl`/`libcrypto` (built by OpenSSL's own `perl Configure`, NOT cmake вҖ?so they DON'T get flag-forwarding above) and `libc++_shared` (straight from the NDK) come out **16 KB on the 64-bit ABIs** (arm64-v8a, x86_64) and **4 KB on the 32-bit ABIs** (armeabi-v7a, x86) вҖ?that's the NDK 27 *default* linker behaviour, and it is exactly correct: Android only requires 16 KB for 64-bit ABIs, so 4 KB on 32-bit is fine. So a healthy build shows a *mixed* table on 32-bit (freerdp*/cjson/uriparser = `0x4000`, ssl/crypto/c++_shared = `0x1000`) and an *all-`0x4000`* table on 64-bit. Verify with `llvm-readelf -l <so> | grep LOAD`, and check the **post-strip** libs that actually ship (extract `lib/<abi>/*.so` from `app-debug.apk`), not just the jniLibs source вҖ?`stripDebugDebugSymbols` preserves alignment but the merged_native_libs step can be re-poisoned by stale copies.

**To rebuild native вҖ?multi-ABI (the current way, added 2026-06):** run **`scripts/build-native-multiarch-in-wsl.sh`** from WSL2 Ubuntu. It defaults to the **three non-arm64 ABIs** (`armeabi-v7a x86 x86_64`) вҖ?arm64-v8a is left untouched because its committed binaries are field-tested (rebuilding risks an ABI-mismatch regression); pass ABIs explicitly (or `ABIS="arm64-v8a armeabi-v7a x86 x86_64"`) to include it. The script loops the ABIs and invokes gradle **once per ABI** with `-PnativeAbi=<abi>` (ONE at a time вҖ?the OpenSSL/FFmpeg ExternalProjects share a source tree and would race on a concurrent multi-ABI build), then stages each ABI's 8 built `.so` + the NDK's `libc++_shared.so` into `jniLibs/<abi>/` and prints a LOAD-alignment table. The single-ABI predecessor `scripts/build-native-in-wsl.sh` still exists (arm64-only). Both install JDK 21 + Linux NDK 27 + cmake via sdkmanager. See `@NATIVE_BUILD_NOTES.md` for the why-not-Windows-native rationale (6 layers of perl/path mismatches in OpenSSL Configure).

Two field-won gotchas baked into the multiarch script:
  - **Download seeding (China-network-fragility mitigation).** ExternalProject re-downloads each dep tarball (openssl 55 MB, ffmpeg 17 MB, cjson, uriparser) into a *per-ABI* `.cxx/.../<abi>/<dep>-prefix/src/` dir, and GitHub HTTP/2 framing errors (`status_code 92`) flake those fetches. The script's `seed_tarballs()` copies the **complete** tarballs from arm64-v8a's existing `.cxx` cache into each new ABI's download dir before building (the generated `download-<dep>.cmake` does "EXISTS + hash match вҶ?skip download"). AGP **wipes a fresh ABI's `.cxx` dir on its first configure**, so a pre-seed is lost there вҶ?the script also **re-seeds + retries once** after a first-attempt failure (by then the dir + CMakeCache exist, so the retry's configure is incremental and the seed survives). Net: download once (or reuse arm64's cache), reuse for every ABI.
  - **FFmpeg x86_64 needs asm OFF.** `ExternalFFmpeg.cmake`: x86_64 is set to `--disable-asm` (like x86 / FFmpeg #4928). FFmpeg's hand-written x86 asm (`libavutil/x86/x86inc.asm`) references global consts (`ff_pw_1`, вҖ? with `R_X86_64_PC32` relocations that `ld.lld` rejects when the static FFmpeg links into the **PIC** shared lib `libfreerdp3.so` (`relocation R_X86_64_PC32 cannot be used against symbol 'ff_pw_1'; recompile with -fPIC`), even with `--enable-pic`. Software decode only вҖ?x86_64 is overwhelmingly emulators, so losing SIMD on the H.264 path is acceptable. arm64 keeps NEON asm (PIC-clean); armeabi-v7a keeps its asm.

While that script runs it backs `local.properties` up to `local.properties.windows-backup` (gitignored) and rewrites `local.properties` to point at the Linux SDK; restores at exit (a `trap`). If `local.properties.windows-backup` shows up in `git status` after an interrupted run, **don't delete it** вҖ?copy it back over `local.properties` to recover the Windows SDK path. (Note: `local.properties` lives on the `/mnt/d` 9p mount, whose Windows-side view lags WSL writes вҖ?if you check it from the Windows side *while* the build is mid-run it may still read the Linux path; trust the post-exit state.)

**Don't attempt Windows-native rebuild** вҖ?every successful patch uncovers a deeper one, abandoned at OpenSSL Configure's out-of-tree path resolution.

## Rendering & input вҖ?load-bearing invariants

- `feature-session/.../render/RdpSurface.kt` is a **plain `View`**, NOT a `SurfaceView`. Earlier SurfaceView + Choreographer + lockCanvas iteration was abandoned вҖ?Compose AndroidView wrap + cross-thread schedule from native worker had too many race modes. Current strategy: `onDraw` re-schedules itself via `scheduleNextFrame()` as long as the view is attached вҖ?either VSync-paced (`postInvalidateOnAnimation`, when the connection's `targetFrameRate == 0` / "auto") or throttled to a fixed fps via `postInvalidateDelayed` (target clamped to the device screen refresh rate). This self-loop must NEVER stop while attached вҖ?it is the load-bearing guard against the "connected but black screen" bug; throttling only DELAYS the next frame, never cancels it. `onAttachedToWindow` kicks one `postInvalidateOnAnimation()` to start the loop (cold-start guard). `markDirty()` is consequently a **no-op** (it must NOT `postInvalidate` вҖ?a fast content-update rate would otherwise bypass the throttle and unfix the frame rate). **The frame source is the double buffer**: `onDraw` reads `RdpSurface.buffer.peekFront()` (the front/complete buffer) every scheduled frame, so a newly committed frame shows up on the next tick. There is no `setBacking` anymore вҖ?pulling from the buffer instead of having Compose push a bitmap per frame keeps the displayed rate decoupled from the content rate (no per-content-frame recompose). Don't reintroduce SurfaceView, and don't route the per-frame bitmap back through Compose state.
- `core-rdp/.../BitmapBuffer.kt` is **double-buffered** (the "йҖҗиЎҢжү«жҸҸ"/tearing fix). The native worker writes the **back** buffer (`nativeBuffer()`), then `commitFrame()` swaps it to the **front** buffer the UI draws вҖ?so the UI (whose hardware-accelerated `drawBitmap` uploads the texture asynchronously on the RenderThread) never reads a buffer being written. `OnGraphicsUpdate` writes the dirty rect into the back buffer, re-applies the previous frame's `staleRect()` (the back buffer is exactly one published generation behind), then `commitFrame()`. The buffer MUST NOT `recycle()` old bitmaps in `resize()`/`release()` вҖ?the UI thread may still hold a reference for 1вҖ? frames; recycle вҶ?`drawBitmap()` hits freed pixel storage вҶ?SIGSEGV, no logcat survives. Let GC reclaim (~2Г—8 MB at 1080p is fine). `current` is updated on resize/release only (drives the "waiting for frame" placeholder), NOT per frame, so SessionCanvas does not recompose at the content rate.
- `feature-session/.../input/ScancodeMap.kt` вҖ?VK constants for extended keys (LWIN/RWIN/APPS/arrows/INSERT/DELETE/HOME/END/PgUp/PgDn/RCTRL/RALT) MUST be pre-ORed with `KBDEXT = 0x0100`. FreeRDP's `GetVirtualScanCodeFromVirtualKeyCode` uses the KBDEXT bit to pick `KBD4T` vs `KBD4X` вҖ?tables don't overlap. Without KBDEXT on an extended-only VK, reverse-lookup returns VK_NONE вҶ?scancode 0 вҶ?server gets nothing. Also: never pass PS/2 scancodes to `sendKeyEvent` directly вҖ?the JNI runs `GetVirtualScanCodeFromVirtualKeyCode(arg, 4)` internally and will double-translate them into garbage.
- `RdpClient.sendMonitorLayout(w, h)` returns `Boolean` and **callers must respect it**. FreeRDP's DRDYNVC Display Control sub-channel comes up several seconds AFTER `OnConnectionSuccess` вҖ?sending before that returns `false` and the PDU is silently dropped. `SessionViewModel.scheduleMonitorLayoutRetry()` runs an exponential backoff (0/0.5/1/2/3/5/5/5/5 s) and stops only when a matching `OnGraphicsResize` arrives (Вұ8 px tolerance for server-side clamping). Don't replace this with a single-shot send.

## App signing

- Both debug and release builds are signed with the same PocketRDP keystore (kept out of git via `.gitignore`). Config lives in `keystore.properties` (also gitignored); see `keystore.properties.template` for the schema. The .jks file is in `keystore/pocketrdp-release.jks`.

## Credentials & encryption

- Connection passwords are encrypted with `core-data/security/CredentialCipher.kt` using Android Keystore alias `pocketrdp_master_v1` (AES/GCM/NoPadding, 256-bit, no user auth required). Plain passwords never hit DB.

## Diagnostics & in-app logging

- `:core-logging` provides `PocketLogger` (Kotlin object singleton, no Hilt). Installed in `PocketRdpApplication.onCreate`. Three sinks: (1) `Log.d/i/w/e` mirror so `adb logcat -s PocketRDP` works; (2) 1000-entry in-memory ring exposed as `StateFlow<List<Entry>>` for `LogScreen` to subscribe; (3) async channel вҶ?`<filesDir>/logs/pocketrdp.log` with 1.5 MB rotate + `.1` backup. The channel decouples native FreeRDP worker thread from file I/O вҖ?DO NOT make the writer synchronous, you'll stall the protocol thread.
- `RdpClient` logs a throttled heartbeat `gfx alive: N frames received` every 2 s (`GFX_LOG_INTERVAL_MS`) while GFX updates flow. In a shared log it distinguishes "picture frozen but still connected" from "network dropped" (both look like a stuck screen) вҖ?the recurring "иҝһдёҠеҮ з§’е°ұж–ӯ/з”»йқўеҶ»з»“" triage class.
- In-app log viewer: `app/src/main/kotlin/.../logs/LogScreen.kt`, navigated to via the icon in `ConnectionListScreen`'s TopAppBar. Filter chips, monospace + colour-by-level, share via `FileProvider` (authority `${applicationId}.fileprovider`, paths in `app/src/main/res/xml/file_provider_paths.xml`, exports `cacheDir/logs/`).
- When users report a bug, ask them to share the log file rather than guess. Crashes that kill the process before the async writer flushes will still lose in-flight entries вҖ?stabilise the crash first (e.g. the BitmapBuffer recycle case above), then ask for logs.
