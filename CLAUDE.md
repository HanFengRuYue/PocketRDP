## Native (FreeRDP) build

**Active via prebuilt jniLibs.** `core-rdp/src/main/jniLibs/arm64-v8a/` ships **9** .so files compiled in WSL2 Ubuntu: `libfreerdp-android` + `libfreerdp3` + `libfreerdp-client3` + `libwinpr3` + `libssl` + `libcrypto` + `libcjson` + `liburiparser` + `libc++_shared`. There is **no** `libopenh264.so` — H.264 comes from **FFmpeg statically linked into `libfreerdp3.so`** (which is why it's ~23 MB). `core-rdp/build.gradle.kts` has `externalNativeBuild` commented out (wrapped in a clearly-marked `/* ===== NATIVE BUILD REGION ===== */` toggle) — Windows-side gradle just packages the prebuilts, no native toolchain needed. **`libcjson.so` and `liburiparser.so` are mandatory** — they are `DT_NEEDED` of `libwinpr3.so`; without them `dlopen(libfreerdp-android.so)` fails on the recursive dependency resolve. The other **8** libs must stay in `packaging { jniLibs { pickFirsts } }` so AGP doesn't drop them at merge time; `libfreerdp-android.so` itself is NOT in pickFirsts (single JNI entry, no duplicate merge source) and doesn't need to be.

**Prebuilt .so feature set**: `WITH_FFMPEG=ON` + `WITH_OPENH264=OFF` (+ `WITH_VIDEO_FFMPEG=ON`, `WITH_DSP_FFMPEG=OFF`); `WITH_OPUS/WEBP/JPEG/PNG=OFF`. **FFmpeg's software h264 decoder + swscale is the only H.264 subsystem**, so `LibFreeRDP.hasH264Support()` returns true and `RdpClient.buildCommandLine` emits **`/gfx:AVC444`** (画质优先, full 4:4:4 chroma) or **`/gfx:AVC420`** (流畅优先, single YUV420 stream — ~half the software-decode cost, no `prim_YUV444` recombine, lower control latency) when `useH264`, chosen by `entity.preferAvc420` (DB v6, default false = AVC444); else `/gfx:RFX`. Never both an AVC mode and RFX — they conflict. AVC420 is also clean with FFmpeg (the magenta/green grid was an OpenH264-only recombine-stride bug, and AVC420 has no recombine). It also always emits `/multitouch` for the rdpei channel, and `/drive:PocketRDP,<path>` when file redirection is on (see Settings).
  - **Why FFmpeg, not OpenH264 (the 卡顿→花屏 saga):** OpenH264 was tried first (WITH_OPENH264=ON, runtime-dlopen mode) and fixed the video 卡顿 against the `AVC444ModePreferred=1` host, but `/gfx:AVC444` rendered a **diagonal magenta/green chroma grid over an otherwise-correct (luma-intact) image** — FreeRDP recombines AVC444's dual streams into YUV444 via its own `prim_YUV`, and OpenH264's decoded plane strides mis-fed that combine. `/gfx:AVC420` (single YUV420 stream, no combine) was clean on OpenH264 but only 4:2:0. Switching the decoder to **FFmpeg** (whose decoded-frame strides feed the combine correctly) made **AVC444 render cleanly, field-confirmed** — so we keep AVC444.
  - **FFmpeg is STATIC** (`ExternalFFmpeg.cmake`: `--disable-shared --enable-static`, minimal `--enable-decoder=h264 --enable-parser=h264` + swscale, no opus/openh264/audio/mediacodec). A shared FFmpeg bakes versioned SONAMEs (`libavcodec.so.61`) into `libfreerdp3` as `DT_NEEDED`, which Android can't package; static links it straight into `libfreerdp3.so` (PIC), no extra `.so`, no soname problem, and 16 KB alignment comes for free from `libfreerdp3`'s own link flags. FreeRDP is told to prefer it via `-DWITH_VIDEO_FFMPEG=ON` in `ExternalFreeRDP.cmake`.
  - **FFmpeg source is NOT vendored** (unlike openh264) — `ExternalFFmpeg.cmake` would download `n8.1.tar.gz` from GitHub. `scripts/fetch-ffmpeg-src.sh` pre-stages it into `cpp/external/ffmpeg/` (with mirror fallbacks) so the WSL build never needs the network mid-build. Run it before re-building native.
  - If FFmpeg ever has to go (size/CPU), the fallback is `WITH_OPENH264=ON` + `WITH_FFMPEG=OFF` + `/gfx:AVC420` (re-add `libopenh264.so` to jniLibs/pickFirsts and the `WITH_OPENH264_LOADING=ON` line in ExternalFreeRDP.cmake — both still present in git history).

Desktop scale uses **`/scale-desktop:<100..300>`** (`FreeRDP_DesktopScaleFactor`, continuous) in `buildCommandLine()` — NOT `/scale`. `/scale` is hard-limited by `parse_scale_options` to {100,140,180}, so the old code quantised to those buckets and every slider value >180 silently collapsed to 180 (field bug: 缩放滑块只有三档真正生效). `/scale-desktop` honours the exact value (it accepts 100..500; we `coerceIn(100,300)` to the slider range), so the 100–300 % slider has real continuous effect. It sets DesktopScaleFactor only (not DeviceScaleFactor like /scale did) and is the **remote** desktop's DPI/font scaling, not local picture zoom. Both are INTEGER options (value required) — note the BOOL-option trap below is a *different* class. When `parse_arguments` fails the snackbar reads "freerdp_parse_arguments failed: Success" because `last_error_code` is only populated during `freerdp_connect`, not during argument parsing — look at the redacted `Connecting with args:` logcat line instead.

**FreeRDP BOOL-option disable trap.** Our args use the slash/colon syntax (`COMMAND_LINE_SEPARATOR_COLON | SIGIL_SLASH | SIGIL_PLUS_MINUS`, no SEPARATOR_SPACE/ENABLE_DISABLE). For a `COMMAND_LINE_VALUE_BOOL` option, *not adding the flag leaves FreeRDP's default* — which for several options is ON. To force-disable you must emit the **`-opt`** form (the `-` sigil → `BoolValueFalse=NULL` → handler `enable=FALSE`). **`/opt:off` FAILS** with `COMMAND_LINE_ERROR_UNEXPECTED_VALUE` (a BOOL option rejects a trailing value), aborting the whole connect. This is why `useMultitransport=false` emits `-multitransport`, not `/multitransport:off` (see the UDP bullet).

**Native touch + transport JNI (added 2026-05).** `LibFreeRDP.java` gained `freerdp_send_touch(inst, contactId, x, y, action)` and `freerdp_get_transport_info(inst)`. The C side: `android_freerdp.h` `androidContext` holds a `RdpeiClientContext* rdpei` (stored on `RDPEI_DVC_CHANNEL_NAME` connect, cleared on disconnect); `android_event.{h,c}` add `EVENT_TYPE_TOUCH` → `android_process_event` calls `rdpei->TouchBegin/Update/End` and forces `rc = TRUE` (a dropped touch when the channel isn't up must NOT tear down the session); `freerdp_get_transport_info` packs `(SelectedProtocol << 1) | (SupportMultitransport && MultitransportFlags!=0 ? 1 : 0)` — bit0 is the "UDP multitransport negotiated" signal (server zeroes `MultitransportFlags` via the `&= remoteFlags` merge in `gcc.c` if it declines UDP). **If you change these signatures, re-run the WSL build** — they live in the prebuilt `.so`.

**16 KB page-size is mandatory** for Android 15+ (SDK 35+) devices and the `sdk_gphone16k_*` emulators — dlopen rejects any .so whose LOAD segments are 4 KB-aligned with `program alignment (4096) cannot be smaller than system page size (16384)`. NDK 27.1 only injects `-Wl,-z,max-page-size=16384` when `ANDROID_SUPPORT_FLEXIBLE_PAGE_SIZES=ON` is passed to cmake, AND ExternalProject children don't inherit init linker flags so each child needs the flag explicitly forwarded. The fix lives in three places and **must** stay synced when re-enabling native build:
1. `core-rdp/build.gradle.kts`: cmake `arguments` block (currently commented inside the disabled native-build region — uncomment alongside the rest).
2. `third_party/FreeRDP/client/Android/Studio/freeRDPCore/src/main/cpp/CMakeLists.txt`: `ANDROID_CMAKE_ARGS` forwards the flags to cjson/uriparser/openssl ExternalProjects (local patch on top of the pinned submodule).
3. `third_party/FreeRDP/client/Android/cmake/ExternalFreeRDP.cmake`: `FREERDP_LINKER_FLAGS` includes `-Wl,-z,max-page-size=16384` because that line explicitly overrides `CMAKE_SHARED_LINKER_FLAGS` for the freerdp3/freerdp-client3/winpr3 child build.

Verify with `llvm-readelf -l <so> | grep LOAD` — every .so's LOAD segments must show `0x4000`, not `0x1000`. (cjson/uriparser produce 16 KB-aligned binaries in their *own* build dirs but the merged_native_libs step can be re-poisoned by stale 4 KB copies in jniLibs/, so always re-verify the files actually under `core-rdp/src/main/jniLibs/arm64-v8a/`.)

**To rebuild native** (e.g. after FreeRDP submodule bump): run `scripts/build-native-in-wsl.sh` from WSL2 Ubuntu. Script installs JDK 21 + Linux NDK 27 + cmake via sdkmanager, runs `:core-rdp:externalNativeBuildDebug` (~13 min after deps cached), then copies `.so` back to jniLibs. See `@NATIVE_BUILD_NOTES.md` for the why-not-Windows-native rationale (6 layers of perl/path mismatches in OpenSSL Configure).

While that script runs it backs `local.properties` up to `local.properties.windows-backup` (gitignored) and rewrites `local.properties` to point at the Linux SDK; restores at exit. If `local.properties.windows-backup` shows up in `git status` after an interrupted run, **don't delete it** — copy it back over `local.properties` to recover the Windows SDK path.

**Don't attempt Windows-native rebuild** — every successful patch uncovers a deeper one, abandoned at OpenSSL Configure's out-of-tree path resolution.

## Hard constraints — don't break these

- **Don't upgrade past AGP 8.13.0 / Hilt 2.55** — AGP 9.x triggers a 3-way deadlock (verified 2026-05): (1) KSP refuses AGP's built-in Kotlin (`KSP is not compatible with AGP built-in Kotlin`, both KSP1 and KSP2 modes); (2) disabling built-in Kotlin (`android.builtInKotlin=false`) + applying `org.jetbrains.kotlin.android` plugin explicitly fails on `BaseExtension cannot be cast` (AGP 9 rewrote the DSL, KGP 2.2.x still references the old base type); (3) Hilt 2.56+ hard-requires AGP 9.0+. Until KSP/Hilt/AGP reconcile, hold AGP 8.13.0 + Hilt 2.55 + Gradle 8.13. Compose BOM 2026.05.01 and Kotlin 2.2.21 *do* work on AGP 8.13.
- **`com.freerdp.freerdpcore.services.LibFreeRDP` class FQN is locked** by `third_party/FreeRDP/.../android_freerdp_jni.h:JAVA_LIBFREERDP_CLASS`. Don't rename or move the file. The shim at `core-rdp/src/main/java/com/freerdp/freerdpcore/services/LibFreeRDP.java` deliberately keeps this path while dropping upstream's BookmarkBase/SessionState/ApplicationSettingsActivity deps.
- **Do NOT re-add `:freeRDPCore` to `settings.gradle.kts`**. Already attempted in commit history — pulls in androidx.appcompat / room 2.8.4 / sqlcipher / preference / recyclerview which conflict with our Compose + Room 2.6.1 stack. The current strategy (compile cpp + minimal LibFreeRDP.java in `:core-rdp`) is deliberate.
- **FreeRDP submodule is pinned at master commit `9b04e3b`** (FreeRDP 3.x). Don't `git submodule update --remote` without verifying the CMake superbuild still works — upstream's `client/Android/cmake/External*.cmake` files have known Windows portability bugs (Unix `:` path separators) that we'd have to re-patch.
- **`app/proguard-rules.pro` must keep `com.freerdp.freerdpcore.**`** — the FreeRDP JNI bridge calls Java callbacks (`OnConnectionSuccess`, `OnGraphicsUpdate`, `OnAuthenticate`, …) via reflection. Without the keep rule, R8 strips them and release builds crash on first connect with `NoSuchMethodError`. The Hilt and Room keep rules in that file are also load-bearing — don't trim "unused" entries.

## Rendering & input — load-bearing invariants

- `feature-session/.../render/RdpSurface.kt` is a **plain `View`**, NOT a `SurfaceView`. Earlier SurfaceView + Choreographer + lockCanvas iteration was abandoned — Compose AndroidView wrap + cross-thread schedule from native worker had too many race modes. Current strategy: `onDraw` re-schedules itself via `scheduleNextFrame()` as long as the view is attached — either VSync-paced (`postInvalidateOnAnimation`, when the connection's `targetFrameRate == 0` / "auto") or throttled to a fixed fps via `postInvalidateDelayed` (target clamped to the device screen refresh rate). This self-loop must NEVER stop while attached — it is the load-bearing guard against the "connected but black screen" bug; throttling only DELAYS the next frame, never cancels it. `onAttachedToWindow` kicks one `postInvalidateOnAnimation()` to start the loop (cold-start guard). `markDirty()` is consequently a **no-op** (it must NOT `postInvalidate` — a fast content-update rate would otherwise bypass the throttle and unfix the frame rate). **The frame source is the double buffer**: `onDraw` reads `RdpSurface.buffer.peekFront()` (the front/complete buffer) every scheduled frame, so a newly committed frame shows up on the next tick. There is no `setBacking` anymore — pulling from the buffer instead of having Compose push a bitmap per frame keeps the displayed rate decoupled from the content rate (no per-content-frame recompose). Don't reintroduce SurfaceView, and don't route the per-frame bitmap back through Compose state.
- `core-rdp/.../BitmapBuffer.kt` is **double-buffered** (the "逐行扫描"/tearing fix). The native worker writes the **back** buffer (`nativeBuffer()`), then `commitFrame()` swaps it to the **front** buffer the UI draws — so the UI (whose hardware-accelerated `drawBitmap` uploads the texture asynchronously on the RenderThread) never reads a buffer being written. `OnGraphicsUpdate` writes the dirty rect into the back buffer, re-applies the previous frame's `staleRect()` (the back buffer is exactly one published generation behind), then `commitFrame()`. The buffer MUST NOT `recycle()` old bitmaps in `resize()`/`release()` — the UI thread may still hold a reference for 1–2 frames; recycle → `drawBitmap()` hits freed pixel storage → SIGSEGV, no logcat survives. Let GC reclaim (~2×8 MB at 1080p is fine). `current` is updated on resize/release only (drives the "waiting for frame" placeholder), NOT per frame, so SessionCanvas does not recompose at the content rate.
- `feature-session/.../input/ScancodeMap.kt` — VK constants for extended keys (LWIN/RWIN/APPS/arrows/INSERT/DELETE/HOME/END/PgUp/PgDn/RCTRL/RALT) MUST be pre-ORed with `KBDEXT = 0x0100`. FreeRDP's `GetVirtualScanCodeFromVirtualKeyCode` uses the KBDEXT bit to pick `KBD4T` vs `KBD4X` — tables don't overlap. Without KBDEXT on an extended-only VK, reverse-lookup returns VK_NONE → scancode 0 → server gets nothing. Also: never pass PS/2 scancodes to `sendKeyEvent` directly — the JNI runs `GetVirtualScanCodeFromVirtualKeyCode(arg, 4)` internally and will double-translate them into garbage.
- `RdpClient.sendMonitorLayout(w, h)` returns `Boolean` and **callers must respect it**. FreeRDP's DRDYNVC Display Control sub-channel comes up several seconds AFTER `OnConnectionSuccess` — sending before that returns `false` and the PDU is silently dropped. `SessionViewModel.scheduleMonitorLayoutRetry()` runs an exponential backoff (0/0.5/1/2/3/5/5/5/5 s) and stops only when a matching `OnGraphicsResize` arrives (±8 px tolerance for server-side clamping). Don't replace this with a single-shot send.
- `feature-session/.../input/SessionGestures.kt` — branches on `InputMode` at gesture start into `awaitTrackpadGesture` (mouse emulation) or `awaitNativeTouchGesture` (RDPEI). **Both MUST release in a `finally`**: trackpad releases a held drag (`if (holdDragActive) controller.endDragHold()`); native-touch releases every still-down contact (`for ((_, slot) in fingerOf) controller.touchUp(...)`). If a finger leaves the view mid-gesture we never see the up event, and without the finally the remote stays with BUTTON1 (or a touch contact) held down forever — the desktop becomes unusable until reconnect. TRACKPAD vocab: tap=left (**two taps = a real remote double-click**), **long-press(400 ms via `withTimeoutOrNull`)→hold-drag** and double-tap→hold-drag (both set `holdDragActive`), 2-finger tap=right, 2-finger vertical=wheel scroll (accumulated in `RdpInputController.scrollAccumPx`, 28 px/notch — a per-event delta is too small to round to a notch otherwise, the old "滚动用不了" bug; gated on a small `twoMove` travel so a still 2-finger tap still falls through to right-click), 3-finger=middle. **There are NO picture-changing gestures.** Per user request the picture is zoomed ONLY via the zoom pill and panned ONLY via the move handle (see SessionScreen below), so every touch gesture is unambiguously a *control* gesture and can't fight the view transform — that is why 2-finger pinch-zoom, double-tap-to-reset, and the auto view-follow were all removed (an earlier round had a `TwoFingerMode` pinch/scroll lock — gone now that there's no pinch), and why a double-tap is a clean remote double-click instead of being eaten to reset the local zoom. NATIVE-TOUCH maps each Compose pointer to a slot 0..9 (= RDPEI externalId) and forwards down/move/up verbatim — no click synthesis, and **no view-follow**: `touchDown/touchMove` call no auto-pan, so a stationary tap's down and up map to the SAME remote pixel (the fix for "触屏缩放后点击无反应" — the old auto-pan shifted the view between down and up so Windows saw a drag, not a tap). Also: `RdpInputController.toggleMode/setMode` defensively call `endDragHold` first so switching modes mid-drag doesn't leak a held button; don't remove that guard.
- `feature-session/.../input/SessionImeBridge.kt` — IME input goes through a **hidden** `BasicTextField` (NOT `BasicTextField2`) anchored by a U+200B zero-width-space sentinel. Most Android IMEs (especially Chinese pinyin) call `InputConnection.commitText()` instead of emitting `KEYCODE_*` events, so the only way to capture them is to host a real text-editing widget. The sentinel keeps the buffer non-empty so backspace behaviour is predictable across vendors; never remove it. `onValueChange` diffs against the sentinel and forwards the fresh substring to `onUnicodeText` → `SessionViewModel.typeText` → `TextInputEncoder` (per-char routing, see next bullet); physical-key paths (Backspace/Enter/arrows/F-keys/modifiers) go through `onPreviewKeyEvent → ScancodeMap.vkFor → sendKeyEvent`. Both paths must coexist — pure-VK would lose Chinese; pure-Unicode would lose Ctrl+A.
- `feature-session/.../input/TextInputEncoder.kt` — committed IME text MUST NOT be blasted out as raw unicode key events. **Sending a unicode keyboard event to a server that did not negotiate `INPUT_FLAG_UNICODE` is fatal to the whole session**: `freerdp_input_send_unicode_keyboard_event` returns FALSE, and `android_event.c:android_process_event` treats *any* input-send failure as grounds to break out of the main connection loop — so the first character typed instantly disconnects (field log: `OnDisconnecting` ~6 ms after the first `sendUnicodeKey`). The encoder routes **printable ASCII → scancode/VK path** (`ScancodeMap.asciiVkFor`, US-QWERTY, Shift-wrapped) which has no unicode dependency, and falls back to unicode only for non-ASCII (CJK/emoji) **gated on `RdpClient.isUnicodeInputSupported()`** — dropping the char (throttled log) rather than disconnecting when the server lacks it. Don't revert `typeText` (in both `SessionViewModel` and `RdpInputController`) to a bare `sendUnicodeKey` loop. Defence-in-depth: `android_event.c` was also patched so the KEY/KEY_UNICODE/CURSOR cases log a warning and force `rc = TRUE` instead of tearing the session down — but that only takes effect after a WSL native rebuild; the Kotlin-side routing is what fixes it on the current prebuilt `.so`.
- `feature-session/.../SessionScreen.kt` — `Modifier.sessionGestures(...)` lives on the **outer Box**, the local zoom/pan `graphicsLayer` (via the `Modifier.userTransformLayer(transformState)` helper) lives on the **inner `AndroidView`** (and on the crosshair `Canvas`, same `transformState`, so they stay in lock-step). PointerInput sits ABOVE graphicsLayer in the modifier chain, so Compose delivers pointer coordinates as the actual on-screen finger position (post-layout, pre-`graphicsLayer`). `RdpInputController.toRemote` therefore inverts BOTH transforms — first the user zoom/pan (`B = C + (S - C - pan)/zoom`, C = view centre) then the fit-to-view scale. (It used to skip the zoom inversion — fine while touch-mode zoom was unused, but the zoom button now magnifies the touch surface in both modes and native-touch needs accurate contact coords, so the full inversion is required. If you ever move pointerInput BELOW graphicsLayer, Compose would pre-apply the transform and you'd have to STOP inverting it here.) The on-screen **zoom pill** (`ZoomControls`: drag up/down = zoom, double-tap = reset to 100% — replaced the old ＋/100%/－ FAB stack, a tall right-edge dead-zone that swallowed 2-finger gestures) and **move handle** (`PanHandle`: shown ONLY while zoomed **AND only in native-touch mode**, drag = pan the magnified picture with gain = current zoom, long-press-then-drag = relocate the handle so it doesn't cover a tap target) mutate `userZoom`/`userPan` — there are deliberately no zoom/pan *gestures*. **TRACKPAD cursor-follow** (`RdpInputController.followCursorWhenZoomed`, called from `drag()`) is the one auto-pan: while zoomed, moving the virtual cursor pans the view via a centred dead-zone (FOLLOW_MARGIN_FRAC=0.25) to keep the cursor visible — restored per user request ("模拟鼠标缩放后跟随鼠标"). It is TRACKPAD-ONLY and safe there because the trackpad cursor is decoupled from the finger, so panning never moves where a click lands; the PanHandle is hidden in TRACKPAD mode precisely because this replaces it. **Native-touch deliberately has NO view-follow** — the old `followRemotePoint` auto-pan was DELETED there because when zoomed it shifted the view between a touch's down and up, so a tap registered as a tiny drag and remote clicks did nothing (that hazard does not apply to the trackpad cursor). So: TRACKPAD = cursor-follow + no handle; native-touch = handle + no follow.
- **Local pinch-zoom/pan plumbing is performance-load-bearing — do NOT regress it.** The transform is owned by `RdpInputController` as a `userTransform: StateFlow<UserTransform>`; every mutator (`setUserZoom`/`setUserPan`/`setViewSize`/`resetUserTransform`) MUST call `publishTransform()` or the on-screen transform freezes mid-gesture. SessionCanvas reads it as `val transformState = controller.userTransform.collectAsStateWithLifecycle()` (kept as `State`, **NOT** `by`) and reads `.value` **only inside the `graphicsLayer {}` lambda** (the `userTransformLayer` helper) — a deferred draw/layer-phase read, so a gesture frame re-records just the GPU layer and never recomposes SessionCanvas. The crosshair overlay likewise reads `virtualPosState.value` inside the `Canvas` draw lambda for the same reason. The earlier design hoisted zoom/pan into recomposing Compose state and mirrored it via a `virtualPosition.collect` sampler — but the pinch path never published `virtualPosition`, so the sampler never woke during a pinch and the `graphicsLayer` stayed frozen until the gesture ended (field symptom: "zoom/pan is very laggy"). Never reintroduce a body-scope `by` read of these flows, and never drop the `publishTransform()` calls. `RdpInputController` keeps the synchronous `userZoom()/userPanX()/userPanY()` getters (the zoom pill, the move handle, and `drag()` sensitivity read them) — only the StateFlow was added on top. `clampPan()` (called from every transform mutator) keeps the scaled framebuffer from being dragged off-screen; it needs the view size, hence `setViewSize` from the Box `onSizeChanged`.
- **Auto-resolution (dynamic resolution) gating.** `SessionViewModel.dynamicResolutionEnabled` is loaded from `entity.dynamicResolution` in `launchConnect` and gates BOTH the `scheduleMonitorLayoutRetry()` call on `Connected` AND `onSurfaceResized` — when the user turned auto-resolution OFF we must send **no** `DISPLAY_CONTROL_MONITOR_LAYOUT` PDUs at all, otherwise the remote desktop still gets resized against the setting (the original "关闭自动分辨率仍被修改" bug — `/dynamic-resolution` was correctly gated in `buildCommandLine`, but the monitor-layout retry/resize path was not). Additionally `onSurfaceResized` skips the send while `imeVisible` is true so the soft keyboard shrinking the local viewport never resizes the remote.
- **Soft-keyboard layout (issues "键盘遮挡画面" + "工具栏随键盘显隐").** The Scaffold content is a `Column`: `SessionCanvas(Modifier.weight(1f))` on top, then the function-key toolbar (`AnimatedVisibility` gated on `state.imeVisible && state.chromeVisible`) below it. The toolbar carries `Modifier.imePadding()`, so its measured height = bar + keyboard height; the `Column`'s `weight(1f)` then shrinks the canvas to sit **above** both the toolbar and the keyboard — that is what lifts the content into view while typing. This deliberately does NOT use `Scaffold.bottomBar`: applying `imePadding()` to a Scaffold slot does NOT grow the content `padding` (Scaffold's `contentWindowInsets` is systemBars-only), so the keyboard would still overlap — a `Column` + `weight` accounts for the sibling's full measured height and is the correct mechanism. The root content `Box` uses `.padding(padding).consumeWindowInsets(padding)` so the toolbar's `imePadding()` nets out the already-consumed navigation-bar inset (no gap above the keyboard). The toolbar is a Column sibling, NOT an overlay, so its taps never leak into the gesture surface. `state.imeVisible` is the single source of truth (toggled by the TopAppBar keyboard button and the toolbar's "收起键盘" button); `SessionImeBridge` shows/hides the IME from it. There is no separate `toolbarVisible` flag anymore. `SessionCanvas` takes a `modifier: Modifier` (the `weight`), not a `PaddingValues`.
- **Immersive system bars + chrome (issue "状态栏一直显示挤占空间").** The Android system bars (status + navigation) are kept **HIDDEN for the whole session** — `SessionScreen` runs `LaunchedEffect(insetsController) { ic.hide(systemBars()) }` with `BEHAVIOR_SHOW_TRANSIENT_BARS_BY_SWIPE` (swipe-revealable), and a `DisposableEffect` re-shows both `systemBars()` and `ime()` on leave. This is now **decoupled** from the app's own top bar: `state.chromeVisible` (renamed from the old `systemBarsVisible`, which controlled BOTH) gates ONLY the `TopAppBar`; the full-screen button toggles chrome, giving two levels — "toolbar + picture" and "pure picture + small exit FAB". Don't re-couple them or gate the `ic.hide(...)` on `chromeVisible`.
- **Display cutout.** `AndroidManifest` sets `android:windowLayoutInDisplayCutoutMode="shortEdges"` (+ `enableEdgeToEdge()` + transparent status bar in the theme). Because the status bar is hidden in-session its inset collapses to 0, so the `TopAppBar` sits flush at the very top (reclaiming the space) — but it is given `windowInsets = WindowInsets.statusBars.union(WindowInsets.displayCutout).only(Horizontal + Top)` so its title/icons still stay clear of the camera notch (top inset in portrait, side inset in landscape/shortEdges) and a transient swipe-reveal doesn't overlap it. The exit FAB and the zoom pill add `.windowInsetsPadding(WindowInsets.displayCutout)` for the same reason. Also `android:windowSoftInputMode="adjustResize"` so Compose `WindowInsets.ime`/`imePadding()` reports the keyboard height for the layout above (the ime inset is independent of system-bar visibility, so hiding the bars doesn't break the soft-keyboard layout).
- **Latency badges (two metrics).** The status dropdown shows **操控延迟** (primary, `state.controlLatencyMs`) and **网络往返** (secondary, `state.latencyMs`).
  - **网络往返** = `RdpClient.measureLatencyMs()` times a throwaway TCP handshake to `host:port` — the raw network path only. A background coroutine samples it every `LATENCY_PROBE_INTERVAL_MS` into `lastLatencyMs`.
  - **操控延迟** = `RdpClient.controlLatencyMs()`, a rolling **low percentile (P25 over the last `CONTROL_LATENCY_SAMPLE_WINDOW`=20 accepted samples)** of the empirical **press→on-screen-frame round-trip** — the lag the user actually FEELS, which is much higher than the TCP RTT because it includes server input-processing, server-side encode, the downstream hop and client H.264 decode (the gdi buffer is already decoded by the time `OnGraphicsUpdate` fires). Measured WITHOUT a native rebuild: `markDiscreteInput()` arms a sample only for a **DISCRETE press** — a click (`sendCursorEvent` with `DOWN|BUTTONn`), a key-down (`sendKeyEvent`/`sendUnicodeKey`), or a touch `DOWN` — AND only on an idle→action edge (`now - lastServerFrameAtMs > CONTROL_LATENCY_IDLE_GAP_MS`). **Cursor MOVES are deliberately NOT armed**: RDP acks a move via a pointer PDU (hardware cursor), NOT a framebuffer update, so timing move→frame paired the move with a much-later unrelated frame (clock tick, popup) and read hundreds of ms on a 0ms-RTT *local* host (field bug: 操控延迟 484ms / 网络往返 0ms). `recordServerFrameForLatency()` (top of `OnGraphicsUpdate`) closes the sample into a recent-sample ring; a gap `> CONTROL_LATENCY_RESPONSE_WINDOW_MS` (**420ms**, tightened from 600) is discarded as "no genuine response" (a press on inert UI that produced no frame), and `markDiscreteInput` also expires a stale pending arm by the same bound so it can't pair late. Reset on `connect()`. **Why P25, not EMA-of-all (the 2026-06 fix for 操控延迟 330ms vs 网络往返 40ms):** the old EMA averaged in samples where a press that caused no visible change was closed by the next *periodic* frame (a terminal's ~500ms cursor blink), inflating the reading ~8×. A genuinely input-caused frame is the *fastest* a frame can follow an input, so the low end of recent samples is the trustworthy estimate; periodic mispairs are always slower and sit in the tail. **P25, not a hard rolling-min** — a blink landing right after a press would otherwise lock in a bogus ~0ms floor. `SessionViewModel` additionally clamps the reported value `≥ lastLatencyMs` (felt latency can't be below the raw RTT). Do **not** add dirty-rect area/intersection gating to reject blink frames — a terminal's single-char echo is also a tiny rect, so it can't be distinguished by size, and "rect must contain the input point" is unreliable in native TOUCH. Under-counts only the final present delay (commit→draw, ≤ one frame). The autodetect RDP RTT would still need a native rebuild AND wouldn't include encode/decode; this client-side proxy avoids the rebuild and is closer to felt latency.
  - Both are folded into UI state by the 1 Hz metrics ticker (still the sole metrics writer).
- **UDP multitransport + transport badge.** `entity.useMultitransport` (DB v3, default ON, auto-fallback to TCP) → `RdpConnectionParams` → `buildCommandLine` emits `/multitransport` when ON and **`-multitransport` when OFF**. The disable form is mandatory: FreeRDP defaults `SupportMultitransport=TRUE` (+ non-zero `MultitransportFlags`) in `settings.c`, so merely omitting the flag is a no-op and UDP is still negotiated (field bug: 关闭 UDP 仍走 UDP). Must be `-multitransport`, never `/multitransport:off` (BOOL-option trap, see above). The actually-negotiated transport is surfaced via `RdpClient.transportInfo()` → `freerdp_get_transport_info` JNI; `SessionViewModel` decodes bit0 into `RdpTransport.{UDP,TCP,UNKNOWN}` in the 1 Hz metrics ticker (UDP comes up a few seconds after Connected, so poll, don't sample once) and the status dropdown shows "网络协议：UDP（多路传输）/ TCP". When adding a `ConnectionEntity` column, bump the DB version and add a `Migration` in `PocketRdpDatabase` (latest is `MIGRATION_5_6`, DB v6 — `prefer_avc420`).
- **Custom fixed resolution.** `entity.customWidth/customHeight` (DB v4, 0/0 = unset). When both > 0, `SessionViewModel.launchConnect` connects at that exact size (replacing the old hardcoded `1920×1080`) AND forces `effectiveDynamic = entity.dynamicResolution && !useCustomRes` so `dynamicResolutionEnabled` is false → NO monitor-layout PDUs resize it. The edit screen's "固定自定义分辨率" switch is mutually exclusive with "动态分辨率" (`toggleCustomResolution` turns dynamic off; the dynamic switch is `enabled = !useCustomResolution`).
- **Per-connection default input mode.** `entity.defaultInputMode` (DB v5, 0 = 模拟鼠标/TRACKPAD, 1 = 直接触屏/TOUCH). `SessionViewModel.launchConnect` sets the initial `state.mode` from it; still toggleable live via the TopAppBar. Edit screen exposes it as two chips (`inputModeOptions`).
- **Settings effectiveness (audited 2026-05, re-audited 2026-06).** Every edit-screen toggle must actually do something. The 2026-06 multi-agent re-audit (all settings end-to-end + adversarial verify) fixed three "looks-set-but-no-op" bugs and corrected stale text:
  - **UDP multitransport** off was a no-op → now emits `-multitransport` (see UDP bullet).
  - **GFX toggle was bypassed when H.264 is on.** `buildCommandLine`'s `when` always emits `/gfx:AVC444` for H.264 (AVC can only run on the rdpgfx channel), so the GFX switch did nothing while H.264 was enabled. The edit screen now **locks the GFX switch ON+disabled while `useH264`** (`checked = useGfx || useH264`, `enabled = !useH264`) and `toggleH264` force-sets `useGfx=true`, so the persisted entity stays honest (no `use_h264=true/use_gfx=false` records). When both are off → no `/gfx` → FreeRDP default `SupportGraphicsPipeline=FALSE` (legacy bitmap); that off-side is correct as-is (GFX is opt-in, no negative flag needed).
  - **H.264 subtitle** said "用 AVC420 硬件编码" — wrong on both counts (we emit `/gfx:AVC444`, decoded by FFmpeg **software**). Corrected to "H.264（AVC444 全彩）…客户端 FFmpeg 软件解码".
  - **剪贴板 text bridge — now BIDIRECTIONAL (text), 2026-06.** Remote→local: `SessionViewModel` injects `@ApplicationContext Context` and handles `RdpEvent.ClipboardReceived` → `setPrimaryClip`; always safe (only writes the Android clipboard, never sends). Local→remote: an `OnPrimaryClipChangedListener` → `rdpClient.sendClipboard` → JNI `EVENT_TYPE_CLIPBOARD`. A shared `lastSyncedClipboard` token guards the echo loop; `clipboardSyncEnabled` (from `entity.redirectClipboard`, loaded in `launchConnect`) gates both directions. **Text only** (plain-string JNI). **`localToRemoteClipboardEnabled = true`** as of the 2026-06 CLEAN FULL WSL rebuild: that rebuild shipped the `android_event.c` `EVENT_TYPE_CLIPBOARD` non-fatal patch (null-check `afc->cliprdr` + force `rc = TRUE`, like TOUCH/KEY) so a clipboard send before the cliprdr channel is up / when the server declined is logged-and-dropped instead of breaking the main loop → disconnect (the old 手机端复制后连接几秒内断开 bug). The rebuild changed **only `libfreerdp-android.so`** (deps stayed byte-identical to the field-tested binaries → no ABI mismatch, avoiding the earlier incremental-rebuild black screen). If you ever rebuild incrementally and ship a mismatched dep set, re-disable this until a CLEAN FULL rebuild. Images would need a native CF_DIB path + rebuild.
  - **文件夹重定向 is now LIVE (2026-06)** via `/drive:PocketRDP,<path>` (rdpdr DRIVE device — the channel is already compiled into the prebuilt `libfreerdp-client3`; **no native rebuild needed**). The path is an **app-private POSIX dir** `getExternalFilesDir(null)/PocketRDP` — a real filesystem path WinPR can `open()/stat()`, NOT the SAF `content://` URI (`entity.sharedFolderUri` stays unused; a SAF passthrough would need a FUSE/temp-copy bridge — a later feature). `SessionViewModel.launchConnect` **`mkdirs()` the dir before connect** — mandatory: FreeRDP's `freerdp_path_valid` silently *skips* a non-existent drive path at parse time (non-fatal, just no drive appears). Disable = omit the flag (`FreeRDP_DeviceRedirection` defaults FALSE, no BOOL-disable trap). The remote sees a "PocketRDP" drive in Explorer; files dropped into the app folder are visible to the host. `entity.soundMode` (0/1/2 → `/sound`,`/microphone`) is deliberately NOT surfaced in the UI: the Android client cpp has no rdpsnd/audin audio backend, so it would be a dead toggle. Don't expose audio as if functional without doing the native work first.
- `SessionViewModel.kt` — FPS and connection duration are written into UI state by a 1 Hz `startMetricsTicker` coroutine, NOT per frame (writing `_state` 30–60×/s would recompose the TopAppBar that often and visibly hitch the framebuffer). The frame counter is ticked from `RdpSurface.onFrameRendered` (one *render* frame, via `SessionViewModel.onFrameRendered`), NOT from the `RdpEvent.GraphicsUpdated` collector — so `fps` reflects the steady render / configured target rate and no longer drops to 0 when the remote screen is idle. The ticker is the sole writer of `fps` / `durationSec` into UI state. Same pattern applies if any other high-frequency metric is added later.

## Common commands

- Debug build: `./gradlew.bat :app:assembleDebug --no-configuration-cache --console=plain --no-daemon` — produces `app/build/outputs/apk/debug/app-debug.apk` (~112 MB, signed with the release keystore via `keystore.properties`; size dominated by bundled FreeRDP+OpenSSL .so, incl. FFmpeg statically linked into `libfreerdp3`).
- Release build: `./gradlew.bat :app:assembleRelease --no-configuration-cache --console=plain --no-daemon` — produces `app/build/outputs/apk/release/app-release.apk` (R8-minified Kotlin, v3-signed; native .so are not shrunk).
- Full clean: `./gradlew.bat clean` (do NOT delete `~/.gradle/caches` — Compose BOM / Hilt artifacts will redownload from Aliyun and take minutes).
- Configuration-cache off (`--no-configuration-cache`) is recommended while iterating on `build.gradle.kts` — it caches aggressively and stale entries cause silent build skips.
- To pull runtime logs without ADB: open the app → main screen TopBar → log icon → share. File lands as `pocketrdp-YYYY-MM-DD.log` via FileProvider, can be sent over QQ/WeChat/email/etc.

## App signing

- Both debug and release builds are signed with the same PocketRDP keystore (kept out of git via `.gitignore`). Config lives in `keystore.properties` (also gitignored); see `keystore.properties.template` for the schema. The .jks file is in `keystore/pocketrdp-release.jks`.
- If `keystore.properties` is missing (fresh clone), `app/build.gradle.kts` falls back to the Android debug signing config so builds still work — but the resulting APK won't match the stable signing identity. Regenerate the keystore with `keytool -genkeypair -keyalg RSA -keysize 2048 -validity 36500 ...` and update the properties file.

## Credentials & encryption

- Connection passwords are encrypted with `core-data/security/CredentialCipher.kt` using Android Keystore alias `pocketrdp_master_v1` (AES/GCM/NoPadding, 256-bit, no user auth required). Plain passwords never hit DB.
- M6 in the original plan was "add encryption" — already done in M1, so skip that milestone label. Plain `password` column does NOT exist; use `passwordCipher: ByteArray` + `passwordIv: ByteArray`.

## Diagnostics & in-app logging

- `:core-logging` provides `PocketLogger` (Kotlin object singleton, no Hilt). Installed in `PocketRdpApplication.onCreate`. Three sinks: (1) `Log.d/i/w/e` mirror so `adb logcat -s PocketRDP` works; (2) 1000-entry in-memory ring exposed as `StateFlow<List<Entry>>` for `LogScreen` to subscribe; (3) async channel → `<filesDir>/logs/pocketrdp.log` with 1.5 MB rotate + `.1` backup. The channel decouples native FreeRDP worker thread from file I/O — DO NOT make the writer synchronous, you'll stall the protocol thread.
- `RdpClient` logs a throttled heartbeat `gfx alive: N frames received` every 2 s (`GFX_LOG_INTERVAL_MS`) while GFX updates flow. In a shared log it distinguishes "picture frozen but still connected" from "network dropped" (both look like a stuck screen) — the recurring "连上几秒就断/画面冻结" triage class.
- In-app log viewer: `app/src/main/kotlin/.../logs/LogScreen.kt`, navigated to via the icon in `ConnectionListScreen`'s TopAppBar. Filter chips, monospace + colour-by-level, share via `FileProvider` (authority `${applicationId}.fileprovider`, paths in `app/src/main/res/xml/file_provider_paths.xml`, exports `cacheDir/logs/`).
- When users report a bug, ask them to share the log file rather than guess. Crashes that kill the process before the async writer flushes will still lose in-flight entries — stabilise the crash first (e.g. the BitmapBuffer recycle case above), then ask for logs.

## What's missing (intentionally, not gaps to fill silently)

No unit tests, no instrumented tests, no CI workflow. detekt (`detekt.yml` + `detekt-baseline.xml`) and `.editorconfig` are in place. Don't add the missing pieces without explicit ask — keep the project minimal until M3+ stabilizes the input/render layer.

detekt is configured with `maxIssues: 0` plus a baseline — any new finding fails the build. The check task is `./gradlew detekt` (there is **no** `detektMain` task — this is one root `detekt {}` over an explicit `source.setFrom` list, not per-module). To accept legitimate new findings (e.g. after a refactor): `./gradlew detektBaseline` regenerates `detekt-baseline.xml`. Prefer that over scattering `@Suppress` annotations.
