## Native (FreeRDP) build

**Active via prebuilt jniLibs ÔÇö FOUR ABIs.** `core-rdp/src/main/jniLibs/{arm64-v8a,armeabi-v7a,x86,x86_64}/` each ship **9** .so files compiled in WSL2 Ubuntu: `libfreerdp-android` + `libfreerdp3` + `libfreerdp-client3` + `libwinpr3` + `libssl` + `libcrypto` + `libcjson` + `liburiparser` + `libc++_shared`. So the APK is a single fat binary carrying all four architectures (Ôëł4├Ś the native payload of the old arm64-only build). There is **no** `libopenh264.so` ÔÇö H.264 comes from **FFmpeg statically linked into `libfreerdp3.so`** (which is why it's ~18ÔÇô23 MB per ABI). **No comment-region toggle anymore:** `core-rdp/build.gradle.kts` is now **property-gated** ÔÇö a normal Windows build (`./gradlew :app:assembleDebug`, no extra props) just packages the four prebuilt ABIs (no native toolchain needed); passing **`-PnativeAbi=<abi>`** turns the FreeRDP CMake superbuild ON for that *one* ABI (WSL2 only). The driver that loops the ABIs is `scripts/build-native-multiarch-in-wsl.sh` (see Native build Ôćĺ multiarch below). **`libcjson.so` and `liburiparser.so` are mandatory** ÔÇö they are `DT_NEEDED` of `libwinpr3.so`; without them `dlopen(libfreerdp-android.so)` fails on the recursive dependency resolve. The other **8** libs (per ABI) must stay in `packaging { jniLibs { pickFirsts } }` so AGP doesn't drop them at merge time (the pickFirsts list is generated for all four ABIs ├Ś 8 libs by a Kotlin loop); `libfreerdp-android.so` itself is NOT in pickFirsts (single JNI entry, no duplicate merge source) and doesn't need to be.

**Prebuilt .so feature set**: `WITH_FFMPEG=ON` + `WITH_OPENH264=OFF` (+ `WITH_VIDEO_FFMPEG=ON`, `WITH_DSP_FFMPEG=OFF`); `WITH_OPUS/WEBP/JPEG/PNG=OFF`. **FFmpeg's software h264 decoder + swscale is the only H.264 subsystem**, so `LibFreeRDP.hasH264Support()` returns true and `RdpClient.buildCommandLine` emits **`/gfx:AVC444`** (šö╗Ŕ┤Ęń╝śňůł, full 4:4:4 chroma) or **`/gfx:AVC420`** (ŠÁüšĽůń╝śňůł, single YUV420 stream ÔÇö ~half the software-decode cost, no `prim_YUV444` recombine, lower control latency) when `useH264`, chosen by `entity.preferAvc420` (DB v6, default false = AVC444); else `/gfx:RFX`. Never both an AVC mode and RFX ÔÇö they conflict. AVC420 is also clean with FFmpeg (the magenta/green grid was an OpenH264-only recombine-stride bug, and AVC420 has no recombine). It also always emits `/multitouch` for the rdpei channel, and `/drive:PocketRDP,<path>` when file redirection is on (see Settings).
  - **Why FFmpeg, not OpenH264 (the ňŹíÚí┐ÔćĺŔŐ▒ň▒Ć saga):** OpenH264 was tried first (WITH_OPENH264=ON, runtime-dlopen mode) and fixed the video ňŹíÚí┐ against the `AVC444ModePreferred=1` host, but `/gfx:AVC444` rendered a **diagonal magenta/green chroma grid over an otherwise-correct (luma-intact) image** ÔÇö FreeRDP recombines AVC444's dual streams into YUV444 via its own `prim_YUV`, and OpenH264's decoded plane strides mis-fed that combine. `/gfx:AVC420` (single YUV420 stream, no combine) was clean on OpenH264 but only 4:2:0. Switching the decoder to **FFmpeg** (whose decoded-frame strides feed the combine correctly) made **AVC444 render cleanly, field-confirmed** ÔÇö so we keep AVC444.
  - **FFmpeg is STATIC** (`ExternalFFmpeg.cmake`: `--disable-shared --enable-static`, minimal `--enable-decoder=h264 --enable-parser=h264` + swscale, no opus/openh264/audio/mediacodec). A shared FFmpeg bakes versioned SONAMEs (`libavcodec.so.61`) into `libfreerdp3` as `DT_NEEDED`, which Android can't package; static links it straight into `libfreerdp3.so` (PIC), no extra `.so`, no soname problem, and 16 KB alignment comes for free from `libfreerdp3`'s own link flags. FreeRDP is told to prefer it via `-DWITH_VIDEO_FFMPEG=ON` in `ExternalFreeRDP.cmake`.
  - **FFmpeg source is NOT vendored** (unlike openh264) ÔÇö `ExternalFFmpeg.cmake` would download `n8.1.tar.gz` from GitHub. `scripts/fetch-ffmpeg-src.sh` pre-stages it into `cpp/external/ffmpeg/` (with mirror fallbacks) so the WSL build never needs the network mid-build. Run it before re-building native.
  - If FFmpeg ever has to go (size/CPU), the fallback is `WITH_OPENH264=ON` + `WITH_FFMPEG=OFF` + `/gfx:AVC420` (re-add `libopenh264.so` to jniLibs/pickFirsts and the `WITH_OPENH264_LOADING=ON` line in ExternalFreeRDP.cmake ÔÇö both still present in git history).

Desktop scale uses **`/scale-desktop:<100..300>`** (`FreeRDP_DesktopScaleFactor`, continuous) in `buildCommandLine()` ÔÇö NOT `/scale`. `/scale` is hard-limited by `parse_scale_options` to {100,140,180}, so the old code quantised to those buckets and every slider value >180 silently collapsed to 180 (field bug: š╝ęŠöżŠ╗ĹňŁŚňĆ¬ŠťëńŞëŠíúšťčŠşúšöčŠĽł). `/scale-desktop` honours the exact value (it accepts 100..500; we `coerceIn(100,300)` to the slider range), so the 100ÔÇô300 % slider has real continuous effect. It sets DesktopScaleFactor only (not DeviceScaleFactor like /scale did) and is the **remote** desktop's DPI/font scaling, not local picture zoom. Both are INTEGER options (value required) ÔÇö note the BOOL-option trap below is a *different* class. When `parse_arguments` fails the snackbar reads "freerdp_parse_arguments failed: Success" because `last_error_code` is only populated during `freerdp_connect`, not during argument parsing ÔÇö look at the redacted `Connecting with args:` logcat line instead.

**FreeRDP BOOL-option disable trap.** Our args use the slash/colon syntax (`COMMAND_LINE_SEPARATOR_COLON | SIGIL_SLASH | SIGIL_PLUS_MINUS`, no SEPARATOR_SPACE/ENABLE_DISABLE). For a `COMMAND_LINE_VALUE_BOOL` option, *not adding the flag leaves FreeRDP's default* ÔÇö which for several options is ON. To force-disable you must emit the **`-opt`** form (the `-` sigil Ôćĺ `BoolValueFalse=NULL` Ôćĺ handler `enable=FALSE`). **`/opt:off` FAILS** with `COMMAND_LINE_ERROR_UNEXPECTED_VALUE` (a BOOL option rejects a trailing value), aborting the whole connect. This is why `useMultitransport=false` emits `-multitransport`, not `/multitransport:off` (see the UDP bullet).

**Native touch + transport JNI (added 2026-05).** `LibFreeRDP.java` gained `freerdp_send_touch(inst, contactId, x, y, action)` and `freerdp_get_transport_info(inst)`. The C side: `android_freerdp.h` `androidContext` holds a `RdpeiClientContext* rdpei` (stored on `RDPEI_DVC_CHANNEL_NAME` connect, cleared on disconnect); `android_event.{h,c}` add `EVENT_TYPE_TOUCH` Ôćĺ `android_process_event` calls `rdpei->TouchBegin/Update/End` and forces `rc = TRUE` (a dropped touch when the channel isn't up must NOT tear down the session); `freerdp_get_transport_info` packs `(SelectedProtocol << 1) | (SupportMultitransport && MultitransportFlags!=0 ? 1 : 0)` ÔÇö bit0 is the "UDP multitransport negotiated" signal (server zeroes `MultitransportFlags` via the `&= remoteFlags` merge in `gcc.c` if it declines UDP). **If you change these signatures, re-run the WSL build** ÔÇö they live in the prebuilt `.so`. **Unicode input force-enable (2026-06).** `android_post_connect` now force-sets `FreeRDP_UnicodeInput=TRUE` right after the capability exchange so CJK/IME text actually reaches the server even when the host doesn't advertise `INPUT_FLAG_UNICODE` (see the TextInputEncoder bullet for the full why). It's a one-line change in `android_freerdp.c` Ôćĺ only `libfreerdp-android.so` changes, so it rebuilt cleanly via `scripts/build-native-multiarch-in-wsl.sh` with all four ABIs passed explicitly (the script default skips arm64-v8a). The OpenSSL/FFmpeg/FreeRDP deps stayed cached, so each ABI was just a freerdp-android recompile + relink ÔÇö the slow part was FreeRDP's own CMake reconfigure on the `/mnt/d` 9p mount (~15 min for the first ABI's `try_compile` sweep), NOT a dep rebuild; the later ABIs reused enough cache to finish in ~6 min each.

**16 KB page-size is mandatory for the 64-bit ABIs** (arm64-v8a, x86_64) on Android 15+ (SDK 35+) devices and the `sdk_gphone16k_*` emulators ÔÇö dlopen rejects any 64-bit .so whose LOAD segments are 4 KB-aligned with `program alignment (4096) cannot be smaller than system page size (16384)`. NDK 27.1 only injects `-Wl,-z,max-page-size=16384` for the *cmake-built* libs (freerdp3/winpr3/freerdp-client3/cjson/uriparser) when `ANDROID_SUPPORT_FLEXIBLE_PAGE_SIZES=ON` is passed to cmake, AND ExternalProject children don't inherit init linker flags so each child needs the flag explicitly forwarded. The fix lives in three places and **must** stay synced:
1. `core-rdp/build.gradle.kts`: cmake `arguments` block inside the `if (nativeAbi != null) { ... }` native-build gate (active only under `-PnativeAbi=<abi>`).
2. `third_party/FreeRDP/client/Android/Studio/freeRDPCore/src/main/cpp/CMakeLists.txt`: `ANDROID_CMAKE_ARGS` forwards the flags to cjson/uriparser/openssl ExternalProjects (local patch on top of the pinned submodule).
3. `third_party/FreeRDP/client/Android/cmake/ExternalFreeRDP.cmake`: `FREERDP_LINKER_FLAGS` includes `-Wl,-z,max-page-size=16384` because that line explicitly overrides `CMAKE_SHARED_LINKER_FLAGS` for the freerdp3/freerdp-client3/winpr3 child build.

**Per-ABI alignment reality (verified 2026-06 against the shipped APK):** `libssl`/`libcrypto` (built by OpenSSL's own `perl Configure`, NOT cmake ÔÇö so they DON'T get flag-forwarding above) and `libc++_shared` (straight from the NDK) come out **16 KB on the 64-bit ABIs** (arm64-v8a, x86_64) and **4 KB on the 32-bit ABIs** (armeabi-v7a, x86) ÔÇö that's the NDK 27 *default* linker behaviour, and it is exactly correct: Android only requires 16 KB for 64-bit ABIs, so 4 KB on 32-bit is fine. So a healthy build shows a *mixed* table on 32-bit (freerdp*/cjson/uriparser = `0x4000`, ssl/crypto/c++_shared = `0x1000`) and an *all-`0x4000`* table on 64-bit. Verify with `llvm-readelf -l <so> | grep LOAD`, and check the **post-strip** libs that actually ship (extract `lib/<abi>/*.so` from `app-debug.apk`), not just the jniLibs source ÔÇö `stripDebugDebugSymbols` preserves alignment but the merged_native_libs step can be re-poisoned by stale copies.

**To rebuild native ÔÇö multi-ABI (the current way, added 2026-06):** run **`scripts/build-native-multiarch-in-wsl.sh`** from WSL2 Ubuntu. It defaults to the **three non-arm64 ABIs** (`armeabi-v7a x86 x86_64`) ÔÇö arm64-v8a is left untouched because its committed binaries are field-tested (rebuilding risks an ABI-mismatch regression); pass ABIs explicitly (or `ABIS="arm64-v8a armeabi-v7a x86 x86_64"`) to include it. The script loops the ABIs and invokes gradle **once per ABI** with `-PnativeAbi=<abi>` (ONE at a time ÔÇö the OpenSSL/FFmpeg ExternalProjects share a source tree and would race on a concurrent multi-ABI build), then stages each ABI's 8 built `.so` + the NDK's `libc++_shared.so` into `jniLibs/<abi>/` and prints a LOAD-alignment table. The single-ABI predecessor `scripts/build-native-in-wsl.sh` still exists (arm64-only). Both install JDK 21 + Linux NDK 27 + cmake via sdkmanager. See `@NATIVE_BUILD_NOTES.md` for the why-not-Windows-native rationale (6 layers of perl/path mismatches in OpenSSL Configure).

Two field-won gotchas baked into the multiarch script:
  - **Download seeding (China-network-fragility mitigation).** ExternalProject re-downloads each dep tarball (openssl 55 MB, ffmpeg 17 MB, cjson, uriparser) into a *per-ABI* `.cxx/.../<abi>/<dep>-prefix/src/` dir, and GitHub HTTP/2 framing errors (`status_code 92`) flake those fetches. The script's `seed_tarballs()` copies the **complete** tarballs from arm64-v8a's existing `.cxx` cache into each new ABI's download dir before building (the generated `download-<dep>.cmake` does "EXISTS + hash match Ôćĺ skip download"). AGP **wipes a fresh ABI's `.cxx` dir on its first configure**, so a pre-seed is lost there Ôćĺ the script also **re-seeds + retries once** after a first-attempt failure (by then the dir + CMakeCache exist, so the retry's configure is incremental and the seed survives). Net: download once (or reuse arm64's cache), reuse for every ABI.
  - **FFmpeg x86_64 needs asm OFF.** `ExternalFFmpeg.cmake`: x86_64 is set to `--disable-asm` (like x86 / FFmpeg #4928). FFmpeg's hand-written x86 asm (`libavutil/x86/x86inc.asm`) references global consts (`ff_pw_1`, ÔÇŽ) with `R_X86_64_PC32` relocations that `ld.lld` rejects when the static FFmpeg links into the **PIC** shared lib `libfreerdp3.so` (`relocation R_X86_64_PC32 cannot be used against symbol 'ff_pw_1'; recompile with -fPIC`), even with `--enable-pic`. Software decode only ÔÇö x86_64 is overwhelmingly emulators, so losing SIMD on the H.264 path is acceptable. arm64 keeps NEON asm (PIC-clean); armeabi-v7a keeps its asm.

While that script runs it backs `local.properties` up to `local.properties.windows-backup` (gitignored) and rewrites `local.properties` to point at the Linux SDK; restores at exit (a `trap`). If `local.properties.windows-backup` shows up in `git status` after an interrupted run, **don't delete it** ÔÇö copy it back over `local.properties` to recover the Windows SDK path. (Note: `local.properties` lives on the `/mnt/d` 9p mount, whose Windows-side view lags WSL writes ÔÇö if you check it from the Windows side *while* the build is mid-run it may still read the Linux path; trust the post-exit state.)

## Rendering & input ÔÇö load-bearing invariants

- `feature-session/.../render/RdpSurface.kt` is a **plain `View`**, NOT a `SurfaceView`. Earlier SurfaceView + Choreographer + lockCanvas iteration was abandoned ÔÇö Compose AndroidView wrap + cross-thread schedule from native worker had too many race modes. Current strategy: `onDraw` re-schedules itself via `scheduleNextFrame()` as long as the view is attached ÔÇö either VSync-paced (`postInvalidateOnAnimation`, when the connection's `targetFrameRate == 0` / "auto") or throttled to a fixed fps via `postInvalidateDelayed` (target clamped to the device screen refresh rate). This self-loop must NEVER stop while attached ÔÇö it is the load-bearing guard against the "connected but black screen" bug; throttling only DELAYS the next frame, never cancels it. `onAttachedToWindow` kicks one `postInvalidateOnAnimation()` to start the loop (cold-start guard). `markDirty()` is consequently a **no-op** (it must NOT `postInvalidate` ÔÇö a fast content-update rate would otherwise bypass the throttle and unfix the frame rate). **The frame source is the double buffer**: `onDraw` reads `RdpSurface.buffer.peekFront()` (the front/complete buffer) every scheduled frame, so a newly committed frame shows up on the next tick. There is no `setBacking` anymore ÔÇö pulling from the buffer instead of having Compose push a bitmap per frame keeps the displayed rate decoupled from the content rate (no per-content-frame recompose). Don't reintroduce SurfaceView, and don't route the per-frame bitmap back through Compose state.
- `core-rdp/.../BitmapBuffer.kt` is **double-buffered** (the "ÚÇÉŔíîŠëźŠĆĆ"/tearing fix). The native worker writes the **back** buffer (`nativeBuffer()`), then `commitFrame()` swaps it to the **front** buffer the UI draws ÔÇö so the UI (whose hardware-accelerated `drawBitmap` uploads the texture asynchronously on the RenderThread) never reads a buffer being written. `OnGraphicsUpdate` writes the dirty rect into the back buffer, re-applies the previous frame's `staleRect()` (the back buffer is exactly one published generation behind), then `commitFrame()`. The buffer MUST NOT `recycle()` old bitmaps in `resize()`/`release()` ÔÇö the UI thread may still hold a reference for 1ÔÇô2 frames; recycle Ôćĺ `drawBitmap()` hits freed pixel storage Ôćĺ SIGSEGV, no logcat survives. Let GC reclaim (~2├Ś8 MB at 1080p is fine). `current` is updated on resize/release only (drives the "waiting for frame" placeholder), NOT per frame, so SessionCanvas does not recompose at the content rate.
- `feature-session/.../input/ScancodeMap.kt` ÔÇö VK constants for extended keys (LWIN/RWIN/APPS/arrows/INSERT/DELETE/HOME/END/PgUp/PgDn/RCTRL/RALT) MUST be pre-ORed with `KBDEXT = 0x0100`. FreeRDP's `GetVirtualScanCodeFromVirtualKeyCode` uses the KBDEXT bit to pick `KBD4T` vs `KBD4X` ÔÇö tables don't overlap. Without KBDEXT on an extended-only VK, reverse-lookup returns VK_NONE Ôćĺ scancode 0 Ôćĺ server gets nothing. Also: never pass PS/2 scancodes to `sendKeyEvent` directly ÔÇö the JNI runs `GetVirtualScanCodeFromVirtualKeyCode(arg, 4)` internally and will double-translate them into garbage.
- `RdpClient.sendMonitorLayout(w, h)` returns `Boolean` and **callers must respect it**. FreeRDP's DRDYNVC Display Control sub-channel comes up several seconds AFTER `OnConnectionSuccess` ÔÇö sending before that returns `false` and the PDU is silently dropped. `SessionViewModel.scheduleMonitorLayoutRetry()` runs an exponential backoff (0/0.5/1/2/3/5/5/5/5 s) and stops only when a matching `OnGraphicsResize` arrives (┬▒8 px tolerance for server-side clamping). Don't replace this with a single-shot send.
- `feature-session/.../input/SessionGestures.kt` ÔÇö branches on `InputMode` at gesture start into `awaitTrackpadGesture` (mouse emulation) or `awaitNativeTouchGesture` (RDPEI). **Both MUST release in a `finally`**: trackpad releases a held drag (`if (holdDragActive) controller.endDragHold()`); native-touch releases every still-down contact (`for ((_, slot) in fingerOf) controller.touchUp(...)`). If a finger leaves the view mid-gesture we never see the up event, and without the finally the remote stays with BUTTON1 (or a touch contact) held down forever ÔÇö the desktop becomes unusable until reconnect. TRACKPAD vocab: tap=left (**two taps = a real remote double-click**), **long-press(400 ms via `withTimeoutOrNull`)Ôćĺhold-drag** and double-tapÔćĺhold-drag (both set `holdDragActive`), 2-finger tap=right, 2-finger vertical=wheel scroll (accumulated in `RdpInputController.scrollAccumPx`, 28 px/notch ÔÇö a per-event delta is too small to round to a notch otherwise, the old "Š╗ÜňŐĘšöĘńŞŹń║ć" bug; gated on a small `twoMove` travel so a still 2-finger tap still falls through to right-click), 3-finger=middle. **There are NO picture-changing gestures.** Per user request the picture is zoomed ONLY via the zoom pill and panned ONLY via the move handle (see SessionScreen below), so every touch gesture is unambiguously a *control* gesture and can't fight the view transform ÔÇö that is why 2-finger pinch-zoom, double-tap-to-reset, and the auto view-follow were all removed (an earlier round had a `TwoFingerMode` pinch/scroll lock ÔÇö gone now that there's no pinch), and why a double-tap is a clean remote double-click instead of being eaten to reset the local zoom. NATIVE-TOUCH maps each Compose pointer to a slot 0..9 (= RDPEI externalId) and forwards down/move/up verbatim ÔÇö no click synthesis, and **no view-follow**: `touchDown/touchMove` call no auto-pan, so a stationary tap's down and up map to the SAME remote pixel (the fix for "ŔžŽň▒Ćš╝ęŠöżňÉÄšé╣ňç╗ŠŚáňĆŹň║ö" ÔÇö the old auto-pan shifted the view between down and up so Windows saw a drag, not a tap). ## Common commands

- Debug build: `./gradlew.bat :app:assembleDebug --no-configuration-cache --console=plain --no-daemon` ÔÇö produces `app/build/outputs/apk/debug/app-debug.apk` (**~204 MB now that it carries all four ABIs**; was ~112 MB arm64-only). Signed with the release keystore via `keystore.properties`; size dominated by the four-ABI FreeRDP+OpenSSL .so, incl. FFmpeg statically linked into each `libfreerdp3`. **Run gradle from the repo root** ÔÇö if the shell's CWD is a subdir like `core-rdp/src/main/jniLibs`, Gradle tries to make that the default project and dies with `Project directory 'ÔÇŽ/jniLibs' is not part of the build defined by settings file`.
- Release build: `./gradlew.bat :app:assembleRelease --no-configuration-cache --console=plain --no-daemon` ÔÇö produces `app/build/outputs/apk/release/app-release.apk` (R8-minified Kotlin, v3-signed; native .so are not shrunk).
- Full clean: `./gradlew.bat clean` (do NOT delete `~/.gradle/caches` ÔÇö Compose BOM / Hilt artifacts will redownload from Aliyun and take minutes).
- Configuration-cache off (`--no-configuration-cache`) is recommended while iterating on `build.gradle.kts` ÔÇö it caches aggressively and stale entries cause silent build skips.
- To pull runtime logs without ADB: open the app Ôćĺ main screen TopBar Ôćĺ log icon Ôćĺ share. File lands as `pocketrdp-YYYY-MM-DD.log` via FileProvider, can be sent over QQ/WeChat/email/etc.

## App signing

- Both debug and release builds are signed with the same PocketRDP keystore (kept out of git via `.gitignore`). Config lives in `keystore.properties` (also gitignored); see `keystore.properties.template` for the schema. The .jks file is in `keystore/pocketrdp-release.jks`.
- If `keystore.properties` is missing (fresh clone), `app/build.gradle.kts` falls back to the Android debug signing config so builds still work ÔÇö but the resulting APK won't match the stable signing identity. Regenerate the keystore with `keytool -genkeypair -keyalg RSA -keysize 2048 -validity 36500 ...` and update the properties file.

## Credentials & encryption

- Connection passwords are encrypted with `core-data/security/CredentialCipher.kt` using Android Keystore alias `pocketrdp_master_v1` (AES/GCM/NoPadding, 256-bit, no user auth required). Plain passwords never hit DB.
- M6 in the original plan was "add encryption" ÔÇö already done in M1, so skip that milestone label. Plain `password` column does NOT exist; use `passwordCipher: ByteArray` + `passwordIv: ByteArray`.

## Diagnostics & in-app logging

- `:core-logging` provides `PocketLogger` (Kotlin object singleton, no Hilt). Installed in `PocketRdpApplication.onCreate`. Three sinks: (1) `Log.d/i/w/e` mirror so `adb logcat -s PocketRDP` works; (2) 1000-entry in-memory ring exposed as `StateFlow<List<Entry>>` for `LogScreen` to subscribe; (3) async channel Ôćĺ `<filesDir>/logs/pocketrdp.log` with 1.5 MB rotate + `.1` backup. The channel decouples native FreeRDP worker thread from file I/O ÔÇö DO NOT make the writer synchronous, you'll stall the protocol thread.
- `RdpClient` logs a throttled heartbeat `gfx alive: N frames received` every 2 s (`GFX_LOG_INTERVAL_MS`) while GFX updates flow. In a shared log it distinguishes "picture frozen but still connected" from "network dropped" (both look like a stuck screen) ÔÇö the recurring "Ŕ┐×ńŞŐňçášžĺň░▒Šľş/šö╗ÚŁóňć╗š╗ô" triage class.
- In-app log viewer: `app/src/main/kotlin/.../logs/LogScreen.kt`, navigated to via the icon in `ConnectionListScreen`'s TopAppBar. Filter chips, monospace + colour-by-level, share via `FileProvider` (authority `${applicationId}.fileprovider`, paths in `app/src/main/res/xml/file_provider_paths.xml`, exports `cacheDir/logs/`).
- When users report a bug, ask them to share the log file rather than guess. Crashes that kill the process before the async writer flushes will still lose in-flight entries ÔÇö stabilise the crash first (e.g. the BitmapBuffer recycle case above), then ask for logs.

## What's missing (intentionally, not gaps to fill silently)

No unit tests, no instrumented tests, no CI workflow. detekt (`detekt.yml` + `detekt-baseline.xml`) and `.editorconfig` are in place. Don't add the missing pieces without explicit ask ÔÇö keep the project minimal until M3+ stabilizes the input/render layer.
